# Ponytail over-engineering review — multi-session (account switching) design

Spec: `docs/superpowers/specs/2026-07-19-multi-session-design.md`
Scope: over-engineering only (correctness/security routed elsewhere).

## Findings

- L66-69 (Log out (this account) action): **simplify**. The spec has the web action call core `/revoke { activeToken }`, *then* conditionally call `/set-active` on another held session. Reading the installed plugin (`node_modules/better-auth/dist/plugins/multi-session/index.mjs`, `revokeDeviceSession` handler) shows `/multi-session/revoke` already does this itself: when the revoked token equals the *active* session's token, the handler looks at the remaining `_multi-` cookies, picks the next valid one, and calls `setSessionCookie` on it in the same response — or clears the session cookie if none remain. That Set-Cookie is already carried home by the existing generic `relaySetCookies`. So the web action is one core call + the existing relay, full stop — no second round-trip, no "if another held session remains" branch to write or test the branching of. What replaces it: a single `POST /revoke` + `relaySetCookies`, same as every other proxied action in this codebase.
- L29-30 / L89-94 (dedicated `/accounts` page + "small link to `/accounts` from `/settings` for discovery"): **simplify**. `/settings` today is one flat `+page.server.ts` (`web/src/routes/settings/+page.server.ts`) with a single named action (`save`) and its own signed-in guard (`hasSession` → `redirect(303,'/')`). SvelteKit supports multiple named actions in one `+page.server.ts` (`?/save`, `?/switchAccount`, `?/logoutAccount`, `?/logoutAll`). Folding the account list + the three mutations into `/settings` as additional sections/actions reuses the existing guard and page shell, and removes both the new route directory *and* the discovery-link line (nothing to discover — it's already the page you're on). This only revisits *where* the UI lives, not decision #4's rejection of a header-menu switcher; if the dedicated-page choice is deliberate for other reasons (route not yet existing when brainstormed, cleaner separation of "who am I" vs "profile fields"), it's fine to keep — flagging because the spec's own review checklist asked for this comparison and the settings page is currently thin enough that folding is the lighter build.
- L36 (`multiSession({ maximumSessions: 3 })`): **minor**. One config value, not exposed as user-facing config (already correctly ruled out in "Out of scope"), so this isn't gold-plating in the sense of a knob nobody asked for — it's a one-line override of the library default (5). Worth a one-line comment on *why* 3 (buffer over the 2 the admin+everyday case needs) if kept, or drop the option entirely and take the library default — either is fine, neither is worth more than a line of diff either way.
- Testing section (L96-110): **keep-as-is**. Already scoped to exactly what the task flagged as mattering: the guest-flow regression guard, the switch/act-as behavior, and (once L66-69 above is simplified) a per-account-logout test that hits one endpoint instead of two. No bullet here re-tests better-auth's own library behavior or adds a speculative case. Nothing to cut.
- "Out of scope" section (L112-117): **keep-as-is**. Explicitly rules out per-user `maximumSessions`, cross-instance sharing, and the client SDK — this is the spec doing its own ponytail pass already. Worth calling out as a positive: the self-restraint here is why the rest of the spec is this short.
- Core change (L34-48, "one line"): **keep-as-is**. `multiSession()` added to the existing plugins array, no migration, no middleware change — `sessionAuth`'s `getSession` call is untouched because the plugin makes the *active* session the one `getSession` returns. Nothing to trim.
- Cookie relay section (L72-85): **keep-as-is**. Correctly identifies that `relaySetCookies`/`cookieHeader` need zero code change (they already relay/forward all cookies generically) and scopes the only real risk (probing `hasSession`'s `session_token` substring match against the plugin's actual `_multi-` cookie naming) as a probe-first step rather than a preemptive rewrite.

## Verdict

Spec is close to minimal already — most sections are "keep-as-is" and the "Out of scope" list shows the author already ran a ponytail pass. One real cut: drop the web-layer "then /set-active" promotion step on per-account logout, since `better-auth`'s `/multi-session/revoke` already promotes the next session server-side in the same response. One optional simplification: consider folding the account list/actions into the existing `/settings` page instead of a new route, since `/settings` is currently a single thin file and SvelteKit's named actions cover the extra mutations without new scaffolding.

net: -1 web round-trip + its branching logic (revoke promotion), possibly -1 route (accounts→settings fold, optional).
